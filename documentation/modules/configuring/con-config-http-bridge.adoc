= Configuring the HTTP Bridge

[role="_abstract"]
Configure your HTTP Bridge deployment by updating the `spec` properties of the `KafkaBridge` custom resource.

Start with a minimal configuration that defines the core requirements.
Extend it with optional settings to support the following:

* Securing connections to Kafka (TLS and authentication)
* Configuring HTTP endpoint settings, such as port and CORS
* Granting the HTTP Bridge access to Kafka resources
* Tuning consumer and producer behavior
* Tuning internal bridge configuration (for example, executor service pool and queue sizes)
* Running multiple HTTP Bridge instances
* xref:con-common-config-{context}[Common configuration], including resource limits and requests (recommended), JVM options, metrics, and pod scheduling
* xref:assembly-logging-configuration-{context}[Logging configuration]

Certain settings, though optional, are recommended for production deployments, such as security and resource allocation.

For details of all configuration options, see the link:{BookURLConfiguring}[Strimzi Custom Resource API Reference^].

== Minimal configuration for the HTTP Bridge

A minimal `KafkaBridge` resource must define:

* `.spec.bootstrapServers`: The bootstrap address for the Kafka cluster.
* `.spec.replicas`: The number of HTTP Bridge pods to deploy.
* `.spec.http.port`: The port on which the HTTP Bridge listens for client requests.

The Kafka cluster does not need to be managed by Strimzi or run on Kubernetes.

.Minimal configuration for KafkaBridge
[source,yaml,subs="+quotes,attributes"]
----
apiVersion: {KafkaBridgeApiVersion}
kind: KafkaBridge
metadata:
  name: my-bridge
spec:
  # Replicas (required)
  replicas: 1
  # Kafka bootstrap servers (required)
  bootstrapServers: my-cluster-kafka-bootstrap:9092
  # HTTP configuration (required)
  http:
    port: 8080
----

* `spec.replicas` specifies the number of HTTP Bridge replicas to run.
* `spec.bootstrapServers` specifies the bootstrap address for the Kafka cluster.
For a Strimzi-managed cluster, you can use the bootstrap service name, for example `<cluster_name>-kafka-bootstrap:<port>`.
* `spec.http.port` specifies the port used by the HTTP Bridge for incoming HTTP requests.

.Example `KafkaBridge` custom resource configuration for TLS encrypted connections to HTTP Bridge server
[source,yaml,subs="+quotes,attributes"]
----
# Basic configuration (required)
apiVersion: {KafkaBridgeApiVersion}
kind: KafkaBridge
metadata:
  name: my-bridge
spec:
  # Replicas (required)
  replicas: 3 # <1>
  # Kafka bootstrap servers (required)
  bootstrapServers: _<cluster_name>_-cluster-kafka-bootstrap:9092 # <2>
  # HTTP configuration (required)
  http: # <3>
    port: 8443
    # HTTP Bridge server TLS configuration (optional)
    tls: # <4>
        # Secret containing server certificate and key (required)
        certificateAndKey:
            secretName: my-bridge-cert-secret
            certificate: cert.crt
            key: key.key
        # SSL configuration (optional)
        config: # <5>
            ssl.enabled.protocols: "TLSv1.1,TLSv1.2,TLSv1.3"
            ssl.enabled.cipher.suites: "TLS_DHE_RSA_WITH_AES_256_GCM_SHA384,TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384"
----
<1> The number of replica nodes.
<2> Bootstrap address for connection to the target Kafka cluster. The address takes the format `<cluster_name>-kafka-bootstrap:<port_number>`. The Kafka cluster doesn't need to be managed by Strimzi or deployed to a Kubernetes cluster.
<3> HTTP access to Kafka brokers.
<4> TLS configuration for encrypted connections between HTTP clients and HTTP Bridge with X.509 certificate and key stored in X.509 in PEM format within the specified secrets.
<5> Additional SSL configurations for HTTP Bridge server.
